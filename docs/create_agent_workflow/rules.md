# 開発ルール (Rules)

プロジェクトを開発・保守する上で、必ず守るべきルールです。

## 1. インフラ・コスト管理
- **完全従量課金制の徹底:** 利用者が少ない時期の維持費を極力ゼロに近づけるため、サーバーレスアーキテクチャ（AWS Lambda, API Gateway, S3, Firebase等）を積極的に採用してください。
- **常時稼働リソースの禁止:** コンテナホスティング（ECS, EKS, Cloud Run等であっても最低課金がある場合）やRDSなどのリレーショナルデータベースは避け、リクエストベースで課金されるサービスを選択してください。

## 2. セキュリティ・プライバシー
- **個人情報の非保持:** ユーザープロフィールには本名、住所、電話番号などの個人識別情報を入力させる項目を設けないでください。すべてニックネーム、アバター画像、任意の自己紹介等で完結する設計とします。
- **情報流出時のリスク最小化:** 万が一データベースへの不正アクセスが発生した場合でも、漏洩する情報が「アイデア」と「ニックネーム」程度にとどまるよう、不必要なデータの収集を行わないでください。
- **API認証の徹底:** すべてのバックエンドAPI（Lambda）のエンドポイントでは、Firebase AuthenticationのIDトークンを検証し、認証済みユーザーからのリクエストのみを受け付けてください。

## 3. 開発環境・技術選定
- **Dockerの利用:** ローカル開発環境の構築には必ずDockerを使用してください（`Dockerfile` および `compose.yaml` を作成）。フロントエンドのNode.js環境や、必要であればLocalStack/Firebase Emulatorなどをコンテナ化します。
- **最新バージョンの使用:** Nuxt.js、各種ライブラリ、AWS SDK、Firebase SDKなどは、原則として最新の安定バージョン（Latest Stable）を利用して環境構築を行ってください。
- **フロント・バックエンドの疎結合:** `service-posting-pages` をフロントエンドとして扱い、バックエンド（API）は独立したデプロイ・管理が可能なように境界を明確に設計してください。

## 4. UI/UX・デザイン
- 既に用意されたフロントエンドのデザインファイル（`service-posting-pages` 内のコード群）を最大限に活用し、必要に応じて動的なデータ連携（API Fetch等）の組み込みを行ってください。
- レスポンシブ対応を意識し、スマートフォンなどのモバイル端末からの投稿や閲覧が快適に行えるよう留意してください。
